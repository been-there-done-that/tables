3 Approaches for the Tables Updater

Evaluating update delivery architecture — pick one to go deep on

A

tauri-plugin-updater + GitHub Releases Recommended

The official Tauri 2 updater plugin. You publish a latest.json manifest to GitHub Releases (or a static URL). The plugin checks it, compares versions, downloads + verifies using Ed25519 signing, and installs. All update transport logic is handled by Tauri.

Pros

  • Minimal code — plugin handles download, verify, install
  • Ed25519 signing built-in (separate from Apple notarization)
  • GitHub Actions can auto-generate the manifest on release
  • Works on macOS, Windows, Linux

Cons

  • Requires generating + storing an Ed25519 keypair as a GitHub secret
  • macOS Gatekeeper warnings without Apple notarization (expected for now)
  • No staged rollouts or A/B without extra tooling
B

tauri-plugin-updater + Custom Update Server

Same plugin, but instead of GitHub Releases JSON you host a small server (Cloudflare Worker, Vercel function, or S3 + Lambda) that returns the update manifest. Binaries still live on GitHub Releases or S3.

Pros

  • Staged rollouts (send update to 10% of users first)
  • Channel support (stable / beta)
  • Update analytics without third parties

Cons

  • Server infra to set up and maintain
  • Overkill for v1 with a small user base
C

DIY — GitHub Releases API via reqwest

Skip the plugin entirely. Call the GitHub Releases API from Rust using your existing reqwest dependency, compare semver, download the asset, verify checksum, and launch the installer. Full custom control.

Pros

  • No new Tauri plugin dependency
  • Complete control over every step
  • Can use SHA256 checksum instead of Ed25519 keypair setup

Cons

  • Must implement: version compare, download progress, verify, install launch
  • More Rust code to maintain
  • Platform-specific installer launch logic (macOS dmg vs Windows msi)
  • No built-in atomic install / rollback
